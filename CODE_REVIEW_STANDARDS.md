## 2. 审查原则

### 2.1 核心原则

| 原则 | 说明 |
|------|------|
| **教育优先** | 审查的目标是提升代码质量和开发者能力，而非指责 |
| **具体明确** | 每个问题必须指明文件路径、行号、修复建议 |
| **解释为什么** | 不只说"改什么"，必须解释"为什么需要改" |
| **建议而非命令** | 使用"建议""考虑"等措辞，除非是 🔴Critical 级别 |
| **分级标注** | 每个问题标注严重等级，便于排优先级 |
| **一次交付** | 单次审查提交完整反馈，不分批点滴式输出 |

### 2.2 不要审查的内容

以下内容由自动化工具覆盖，人工审查不必关注：
- 代码风格（缩进、空格、换行）→ ESLint / Checkstyle
- 导入排序 → IDE 自动优化
- 简单的命名规范 → IDE 静态检查
- 已知的 false positive → 在审查报告中标注 `### 已知误报`

## 3. 严重等级分类

### 3.1 等级定义

| 等级 | 标记 | 定义 | 处理时限 | 合并策略 |
|------|------|------|---------|---------|
| **Critical** | 🔴 | 系统崩溃、安全绕过、数据丢失风险 | 立即修复 | **阻止合并** |
| **High** | 🟠 | 功能异常、数据不一致、严重后果的潜在漏洞 | 24小时内 | **阻止合并** |
| **Medium** | 🟡 | 逻辑缺陷、性能退化、非关键路径的安全问题 | 本周内 | 建议修复后合并 |
| **Low** | 🟢 | 代码质量、风格、微弱性能影响 | 下个迭代 | 可合并后修复 |
| **Nit** | 💭 | 建议性改进、可读性微调、替代方案 | 无需承诺 | 作者自行决定 |

### 3.2 等级判定矩阵

```
是否导致系统崩溃/数据丢失/安全绕过？
  ├── 是 → 🔴 Critical
  └── 否 → 是否导致核心功能异常或数据不一致？
            ├── 是 → 🟠 High
            └── 否 → 是否增加技术债务或降低可维护性？
                      ├── 是 → 🟡 Medium
                      └── 否 → 是否影响代码美感或可读性？
                                ├── 是 → 🟢 Low
                                └── 看心情 → 💭 Nit
```

### 3.3 本项目专属判定标准

| 场景 | 自动等级 |
|------|---------|
| `@CacheEvict` / `@Cacheable` 注解误用 | 🟠 High（已出过 Critical bug） |
| 缺少 `@PreAuthorize` 的敏感接口 | 🔴 Critical |
| JWT/Token 相关逻辑错误 | 🔴 Critical |
| 前后端 API 契约不一致 | 🟠 High |
| 未检查 SQL UPDATE 返回行数 | 🟡 Medium |
| 缺少乐观锁/分布式锁的场景 | 🟠 High |
| Promise 错误未 catch | 🟡 Medium |
| `undefined`/`null` 未防御 | 🟡 Medium |
| 硬编码密码/密钥 | 🔴 Critical |
| N+1 查询 | 🟡 Medium |

## 4. 审查流程

### 4.1 流程总览

```
开发者提交 PR
      │
      ▼
┌──────────────────────────────────────────────┐
│            Stage 1: 自动化门禁                │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐         │
│  │  CI检查  │ │ 覆盖率   │ │ 构建验证 │         │
│  │ Checkstyle│ │ ≥70%   │ │ 编译成功 │         │
│  │ ESLint   │ │         │ │         │         │
│  └────┬─────┘ └────┬────┘ └────┬────┘         │
│       │             │            │             │
│       └──────┬──────┴────────────┘             │
│              ▼                                 │
│         全部通过 → Stage 2                     │
│         任一失败 → 拒绝，提醒开发者修复          │
└──────────────────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────────────┐
│            Stage 2: 人工审查                  │
│                                               │
│  审查者: 同模块负责人或指定 Reviewer            │
│  审查维度: 见 §5                               │
│  审查时间: 小型PR(<200行) → 30min内 返回       │
│           中型PR(200-800行) → 2小时 内返回      │
│           大型PR(>800行) → 拆分提交，拒绝大PR   │
│                                               │
│  审查者关注点:                                 │
│  1. 正确性 — 逻辑是否正确？                     │
│  2. 安全性 — 是否有安全漏洞？                   │
│  3. 可维护性 — 6个月后能看懂吗？                 │
│  4. 性能 — 是否有明显瓶颈？                     │
│  5. 测试 — 关键路径有测试覆盖吗？                │
└──────────────────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────────────┐
│            Stage 3: 反馈与迭代                │
│                                               │
│  审查者: 提交审查意见（使用 §7 模板）           │
│  开发者: 逐条修复或回复理由                     │
│                                               │
│  🔴 Critical / 🟠 High:                        │
│     → 必须修复，修复后重新 Request Review       │
│                                               │
│  🟡 Medium:                                    │
│     → 建议修复，如暂不修复需添加 TODO            │
│                                               │
│  🟢 Low / 💭 Nit:                              │
│     → 作者自行决定，审查者不阻塞合并             │
└──────────────────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────────────┐
│            Stage 4: 最终验证                  │
│                                               │
│  审查者确认所有 🔴🟠 已修复                     │
│  → Approve PR                                 │
│  自动化: 合并前再次运行 CI                      │
│  → 通过后自动合并                              │
└──────────────────────────────────────────────┘
```

### 4.2 角色与职责

| 角色 | 职责 | 本项目人员 |
|------|------|----------|
| **PR 作者** | 自审自查、编写测试、修复反馈 | 所有开发者 |
| **Reviewer** | 审查代码正确性/安全性/可维护性 | 模块负责人 |
| **Security Reviewer** | 专项审查安全漏洞（涉及认证/授权/数据的PR） | 安全负责人 |
| **Tech Lead** | 最终审批、架构决策 | 项目负责人 |

### 4.3 PR 规模控制

```
PR 规模限制:
├── 推荐: 100-400 行
├── 上限: 800 行（超出必须拆分）
└── 例外: 纯重构（无逻辑变化）可放宽至 1200 行
```

## 5. 审查维度与检查清单

### 5.1 通用检查清单（所有 PR 适用）

每项审查必须覆盖以下维度，标记 ✅（通过）/ ❌（有问题）/ N/A（不适用）：

| # | 检查项 | 维度 | 说明 |
|---|--------|------|------|
| 1 | 功能正确性 | 正确性 | 代码是否实现了预期功能？是否有遗漏的边界条件？ |
| 2 | 安全漏洞 | 安全 | OWASP Top 10 是否已覆盖？用户输入是否校验？ |
| 3 | 权限控制 | 安全 | 后端接口是否有合适的 `@PreAuthorize`？ |
| 4 | 错误处理 | 健壮性 | 异常是否正确捕获和处理？是否有吞异常？ |
| 5 | 空值防护 | 健壮性 | `null`/`undefined` 是否做了防御性检查？ |
| 6 | 并发安全 | 正确性 | 共享状态是否有锁保护？乐观锁是否覆盖？ |
| 7 | 缓存一致性 | 正确性 | `@CacheEvict`/`@Cacheable` 使用是否正确？ |
| 8 | 数据完整性 | 正确性 | SQL UPDATE/DELETE 返回行数是否检查？ |
| 9 | 日志记录 | 可维护性 | 关键操作是否记录日志？敏感信息是否脱敏？ |
| 10 | 测试覆盖 | 测试 | 核心逻辑是否有单元测试？Happy Path + Error Path？ |
| 11 | 命名清晰 | 可维护性 | 变量/函数/类命名是否自解释？ |
| 12 | 文档注释 | 可维护性 | 复杂逻辑是否有注释？公开 API 是否有 JSDoc/JavaDoc？ |
| 13 | API 契约 | 正确性 | 前后端 DTO/接口定义是否一致？ |
| 14 | 性能影响 | 性能 | 是否有 N+1 查询？大数据量分页是否正确？ |
| 15 | 依赖变更 | 可维护性 | 是否引入了新的第三方依赖？许可证是否兼容？ |

### 5.2 安全专项检查（涉及认证/授权/敏感数据的 PR）

| # | 检查项 | 说明 |
|---|--------|------|
| S1 | 认证绕过 | Token 类型是否校验？Refresh Token 能当 Access Token 用吗？ |
| S2 | IDOR | 资源访问是否校验了所有权？ |
| S3 | 输入注入 | SQL 注入？XSS？命令注入？ |
| S4 | 密钥管理 | 密码/密钥/Token 是否有硬编码？是否从环境变量读取？ |
| S5 | 敏感数据暴露 | 日志/响应中是否包含 Token、密码、手机号等敏感信息？ |
| S6 | CORS/CSRF | CORS 配置是否过宽？CSRF Token 是否正确实现？ |
| S7 | 密码策略 | 默认密码是否为弱密码？密码强度校验是否存在？ |

## 6. 模块专项检查清单

### 6.1 后端 — Controller 层

```
□ @PreAuthorize / @RolesAllowed 注解是否覆盖所有敏感接口？
□ 请求参数校验（@Valid/@NotBlank 等）是否完整？
□ 返回值是否使用了统一的 Result<T> 包装？
□ 分页接口是否限制了最大 pageSize？
□ 敏感操作（删除、修改权限等）是否有操作日志？
□ 导出/批量操作是否有时限和数量上限？
```

### 6.2 后端 — Service 层

```
□ 事务边界是否正确（@Transactional 注解位置）？
□ @Cacheable/@CacheEvict 注解参数是否正确？
   □ beforeInvocation 是否正确设置？
   □ key 表达式是否使用了 #result 而非入参？
   □ 是否同时考虑了本地缓存（Caffeine）和远程缓存（Redis）？
□ 乐观锁（version 字段）是否在 UPDATE 后检查返回行数？
□ 分布式锁（Redisson）的租约时间是否足够，是否考虑了 GC 暂停？
□ 异常是否正确转换为业务异常（非 RuntimeException 裸抛）？
□ 幂等性是否得到保证（重复提交是否有保护）？
```

### 6.3 后端 — Mapper / MyBatis

```
□ SQL 是否使用了参数化查询（防止 SQL 注入）？
□ 动态 SQL（<if>/<choose>）是否有默认值？
□ 大批量查询是否有分页？是否使用了游标？
□ UPDATE/DELETE 是否检查 affected rows？
□ 索引是否在 SQL 中被有效使用？
```

### 6.4 后端 — 配置与基础设施

```
□ application.yml 中的敏感配置是否通过环境变量注入？
□ Nginx 配置中 return/rewrite 指令位置是否正确（不在 location 块之前）？
□ Docker Compose 中服务依赖（depends_on）是否配置了健康检查？
□ Redis 连接是否设置了密码认证？
□ CORS 配置中 allowCredentials=true 时，allowedOrigins 是否未使用 "*"？
```

### 6.5 前端 — API 层（utils/request.js）

```
□ Token 刷新逻辑是否正确处理并发请求？
□ 错误响应中的 data.message 是否有兜底值（非 "undefined"）？
□ 401/403 响应是否统一跳转登录页？
□ CSRF Token 是否正确携带？
□ 请求缓存/去重是否有合理的过期策略？
```

### 6.6 前端 — Pinia Store

```
□ 是否只有一个 Pinia 实例（避免孤立实例）？
□ 持久化插件（pinia-plugin-persistedstate）是否正确配置？
□ State 更新是否通过 actions（而非直接修改）？
□ 敏感状态（Token、密码）是否使用 sessionStorage 而非 localStorage？
```

### 6.7 前端 — Vue 组件

```
□ 是否避免了 v-if 和 v-for 同时使用？（如无法避免，是否用 <template> 包裹 v-if？）
□ 异步数据加载是否有 loading/error/empty 三种状态？
□ 表单提交是否有防重复提交保护？
□ 对话框取消/关闭是否都做了正确处理？
□ computed 和 watch 是否有正确清理（watch 的 immediate/deep 选项是否必要）？
□ 是否使用了 BroadcastChannel 处理多标签页同步（而非 Cookie）？
```

### 6.8 前端 — 路由

```
□ 路由守卫是否覆盖所有需要认证的页面？
□ 动态路由权限是否在 token 过期后正确处理？
□ 404/无权限页面是否有独立的处理？
□ 页面切换是否有 NProgress 或其他进度指示？
```
